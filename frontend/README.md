# Purple UI

One React app, two ways to run it — no duplicated code:

- **Web UI (default):** built with **Vite** into `../src/purple/web/`, which the Python
  backend serves at **http://127.0.0.1:8765/ui/** and opens automatically on startup.
  Needs only Node.js.
- **Native desktop app (optional):** the same app wrapped by **Tauri** in its own window.
  Needs Node.js **and** the Rust toolchain.

The React source lives in `ui-src/` (`App.tsx`, `purple.ts`, `App.css`, `main.tsx`). Both
targets build from it — the intelligence all lives in the Python backend; this is just the face.

## Web UI

```bash
cd frontend
npm install
npm run build       # outputs to ../src/purple/web  ->  served at /ui/
# or, for live development with hot reload against the running backend:
npm run dev         # http://localhost:5173
```

`setup.ps1` runs `npm install` + `npm run build` for you (installing Node via winget if
needed), so after setup the web UI just works and auto-opens.

## Native desktop app (Tauri)

```bash
# from the repo root:
powershell -ExecutionPolicy Bypass -File scripts\build_tauri.ps1
```

That installs Rust if missing, scaffolds `src-tauri/` the first time (wired to this Vite
project), and launches the window in dev mode. For an installable `.exe`:

```bash
cd frontend
npm run tauri build
```

If `tauri init` rejects a flag on your CLI version, run `npm run tauri init` interactively
and answer: dev command `npm run dev`, build command `npm run build`, dev URL
`http://localhost:5173`, web assets `../../src/purple/web`.

## Notes

- The client talks to the backend at `http://127.0.0.1:8765` (and `ws://.../ws`). Served
  from the backend it's same-origin, so there are no CORS issues.
- `base: "./"` in `vite.config.ts` keeps asset paths relative, so the same build works both
  under the `/ui/` sub-path and inside Tauri's `file://` loader.
- The built output (`src/purple/web/`) and `node_modules/` are git-ignored — regenerated by
  the build.

## Backend API contract

- `GET /health` -> `{ ok, ollama, model, ... }`
- `POST /chat` -> `{ "message": str, "session_id"?: str }` -> `{ "reply": str }`
- WebSocket `/ws`: send `{ "message": "..." }`; may receive
  `{ "type": "confirm_request", "tool", "args" }` -> reply `{ "approved": bool }`;
  streams `{ "type": "reply_chunk", "text" }` then `{ "type": "reply", "reply" }`.
- `GET/POST /observe`, `GET /suggestions`, `GET/POST /config`, plus the
  automations / memory / documents / missions endpoints.
